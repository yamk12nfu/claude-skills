# 要件定義書作成スキル

ユーザーとの対話を通じて曖昧なアイデアを実装可能な要件定義書に仕上げるスキル。

## ワークフロー

### 1. ヒアリング開始

ユーザーの最初の入力から、プロジェクトの概略を把握する。構成要素（UI / API / バッチ / 業務自動化等）を特定する。複合構成（例: Web UI + API + バッチ）が一般的なので、複数選択を前提とする。

### 2. 段階的ヒアリング

[questioning-guide.md](references/questioning-guide.md)を参照し、以下の順序で質問を進める:

1. **プロジェクト概要・背景・目的** — なぜ作るのか、構成要素は何か
2. **ターゲットユーザー・利用主体** — 誰が（何が）どう使うのか
3. **機能棚卸しと優先度付け** — 全体像を把握し、Must/Should/Couldに分類
4. **Must機能の詳細深掘り** — I/O、バリデーション、エラー、インターフェース要件
5. **非機能要件** — 性能、セキュリティ、可用性、運用
6. **技術要件** — 技術スタック（未定でも可）

#### ヒアリングのルール

- 一度に出す質問は必要な数だけ。多すぎない。
- 曖昧な回答は具体化するまで掘り下げる（「適切に」→具体的な条件に）。
- ユーザーの回答から推測できることは確認を取りつつ補完する。
- **まず全機能を洗い出し、優先度を付けてからMust機能の詳細に入る。**
- Should/Could機能は概要レベルの記載に留め、詳細深掘りはMust機能に集中する。
- 機能の詳細深掘りでは、構成要素に応じた確認項目を使い分ける:
  - UI → 画面要件（レイアウト、遷移、インタラクション）
  - API → エンドポイント、リクエスト/レスポンス、認証、レート制限
  - バッチ → 実行トリガー、入出力ソース、リトライ、冪等性
  - 業務自動化 → 対象システム、操作手順、前提条件、失敗時の再開方法、人間の介入ポイント
- 「他にありますか？」で漏れを防ぐ。
- 「〜の場合は？」でエッジケースを洗い出す。

### 3. ヒアリング完了判断

以下をすべて満たしたらヒアリング完了:
- プロジェクトの目的・背景・構成要素が明確
- ターゲット（ユーザー / 外部システム / スケジューラ等）と主要な利用シーンが特定済み
- 機能の全体像と優先度が整理済み
- Must機能について以下が定義可能なレベル:
  - 共通: I/O・バリデーション・エラーハンドリング・データ永続化
  - UIを含む機能: 画面要件（レイアウト、遷移、インタラクション）
  - APIを含む機能: エンドポイント、認証、レート制限
  - バッチを含む機能: 実行トリガー、リトライ、冪等性
  - 業務自動化を含む機能: 対象システム、操作手順、前提条件、失敗時の再開方法、人間の介入ポイント
- 非機能要件が構成要素に応じて確認済み（性能・セキュリティ・可用性・運用）
- 技術スタックは確認済み（未定でも可）

完了時は「ヒアリング内容をもとに要件定義書を作成します。よろしいですか？」と確認する。

ユーザーが途中で「もういい」「これで作って」と言った場合は、得られた情報の範囲で要件定義書を作成し、未確定事項は「未確定事項一覧」セクションにまとめる。

### 4. 要件定義書の生成

[document-template.md](references/document-template.md)のテンプレートに沿って要件定義書を生成する。プロジェクトの構成要素に応じて該当するセクションのみ含める。

生成ルール:
- ヒアリングで得た情報を正確に反映する。
- 要求仕様にはID・受け入れ条件を含める。人間ユーザー向けはユーザーストーリー形式、システム連携・バッチ・業務自動化はシステム要求形式を使う。
- 機能要件はI/O表・バリデーション表・エラーハンドリング表を統一的に表形式で記載する。
- 画面遷移図はMermaidで記述する（UIを含む場合のみ）。
- 推測で補完した箇所や未確定の項目は「未確定事項一覧」セクションに集約する（本文中に散在させない）。
- 技術スタックは選定理由を含める（未定の場合は「未定」と明記）。
- 構成要素に存在しないセクションは出力しない（例: UIがなければ画面構成セクションを丸ごと省略）。

### 5. レビューと修正

生成後、ユーザーにレビューを依頼する。指摘があれば修正し、要件定義書を更新する。
