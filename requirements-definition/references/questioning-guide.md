# ヒアリング質問ガイド

各フェーズで確認すべき観点と質問例。フェーズ順に進めるが、会話の流れに応じて柔軟に前後して良い。

## フェーズ1: プロジェクト概要・背景・目的

**ゴール**: なぜ作るのか、何を解決するのか、どんな構成要素を持つかを明確にする

確認観点:
- 現状の課題・ペインポイント
- このプロジェクトで実現したいこと
- 構成要素の特定（UI / API / バッチ / 業務自動化 — 複数可）
- 既存システムの有無（リプレイスか新規か）
- ビジネス上の制約（予算、期限、チーム規模）
- 成功の定義（KPI・ゴール指標）

質問例:
- 「このプロジェクトが必要になった背景を教えてください」
- 「現在はどのように業務を行っていますか？何が課題ですか？」
- 「このシステムにはどんな要素がありますか？（例: ユーザーが操作するWeb画面、外部向けAPI、定期実行バッチなど、複数あれば全て教えてください）」
- 「既存のシステムやツールはありますか？」
- 「プロジェクトが成功したと言える状態はどのようなものですか？」

## フェーズ2: ターゲット・利用主体

**ゴール**: 誰が（何が）どんな状況で使うのかを具体化する。利用主体は人間に限らない。

確認観点:
- 利用主体の種類（管理者、一般ユーザー、外部システム、スケジューラ等）
- 各利用主体の種別（人間 / システム / タイマー）
- 各利用主体の主な目的・タスク
- 利用頻度・利用環境（PC/モバイル、社内/社外）※人間ユーザーの場合
- ユーザーのITリテラシー ※人間ユーザーの場合
- 同時利用数の想定

質問例:
- 「このシステムを使うのはどんな人たちですか？また、連携する外部システムはありますか？」
- 「それぞれの利用主体が主にやりたいこと（または期待する処理）は何ですか？」
- 「PCとスマートフォンどちらからの利用が多いですか？」（人間ユーザーの場合）
- 「同時にどのくらいの人数・リクエスト数が想定されますか？」

## フェーズ3: 機能棚卸しと優先度付け

**ゴール**: 全体の機能を洗い出し、Must/Should/Couldに分類する。詳細に入る前にスコープの全体像を確定させる。

確認観点:
- 必要な機能の全体像
- 各機能がどの構成要素に属するか（UI / API / バッチ等）
- 各機能の優先度（Must/Should/Could）
- MVPに含める範囲
- フェーズ分けの要否

質問例:
- 「必要な機能を思いつく限り挙げてください」
- 「それぞれの機能はどの構成要素に関連しますか？（画面操作、API、バッチなど）」
- 「それぞれの機能について、初期リリースに必須か、あると嬉しいか、将来対応でよいか教えてください」
- 「最小限の構成（MVP）で動かすなら、どの機能が絶対に必要ですか？」
- 「他に漏れている機能はありませんか？」

**このフェーズが完了してから、フェーズ4のMust機能の詳細深掘りに入る。**

## フェーズ4: Must機能の詳細深掘り

**ゴール**: Must機能のI/O、バリデーション、エラーハンドリング、インターフェース要件を定義できるレベルまで具体化する。構成要素に応じた確認項目を使い分ける。

確認観点（共通）:
- 各機能の入力項目と出力
- バリデーションルール（文字数、形式、必須/任意）
- 異常系の振る舞い（エラー時にどうなるべきか）
- データの永続化要否と保持期間

確認観点（UIを含む機能）:
- 画面のレイアウトイメージ
- 画面間の遷移
- 一覧表示時のソート・フィルタ・ページネーション
- 検索条件

確認観点（APIを含む機能）:
- エンドポイント設計
- リクエスト/レスポンス形式
- 認証方式
- レート制限

確認観点（バッチを含む機能）:
- 実行トリガー（cron、イベント等）
- 入力ソースと出力先
- リトライポリシー（回数、間隔、条件）
- 冪等性の担保方法

確認観点（業務自動化を含む機能）:
- 対象システム（RPA対象アプリ、SaaS等）
- 操作手順（ステップバイステップ）
- 前提条件（ログイン状態、データの事前準備等）
- 失敗時の再開方法（途中再開 / 最初からやり直し）
- 人間の介入が必要なポイント

質問例:
- 「{機能X}について、入力する情報は何がありますか？」
- 「{項目Y}に入力制限はありますか？（文字数、形式など）」
- 「{機能X}でエラーが起きた場合、どうなるべきですか？」
- 「{画面Z}にはどんな情報が表示されますか？」（UIの場合）
- 「このAPIのレスポンスにはどんなフィールドが必要ですか？レート制限は必要ですか？」（APIの場合）
- 「このバッチが途中で失敗した場合、リトライしますか？その場合の冪等性はどう担保しますか？」（バッチの場合）
- 「自動化の対象システムは何ですか？操作手順を教えてください」（業務自動化の場合）
- 「途中で失敗した場合、途中から再開できますか？それとも最初からやり直しですか？」（業務自動化の場合）
- 「このデータはどこに保存しますか？保持期間はありますか？」

**深掘りのコツ**:
- ユーザーが「〜機能」と言ったら、その機能を使う具体的なシナリオを聞く
- 「他にはありますか？」で漏れを防ぐ
- 「〜の場合はどうなりますか？」でエッジケースを洗い出す
- 曖昧な表現（「適切に」「うまく」等）は具体的な条件に落とし込む

## フェーズ5: 非機能要件

**ゴール**: 性能、セキュリティ、可用性、運用要件を確認する

確認観点（共通）:
- セキュリティ要件（認証方式、権限管理、データ保護）
- 可用性（SLA、障害対応）
- 運用要件（ログ、監視、バックアップ、CI/CD）

確認観点（UI / APIを含む場合）:
- レスポンスタイム
- 同時接続数・スループット

確認観点（バッチ / 業務自動化を含む場合）:
- 処理完了期限（SLA）
- 実行ウィンドウ（許容される実行時間帯）
- データ鮮度要件（どの時点のデータが必要か）
- RTO/RPO（障害復旧目標）

質問例:
- 「レスポンスタイムに要件はありますか？」（UI/APIの場合）
- 「このバッチ処理はいつまでに完了している必要がありますか？実行可能な時間帯に制約はありますか？」（バッチの場合）
- 「障害時、何分以内に復旧する必要がありますか？データはどの時点まで戻せればよいですか？」（バッチ/自動化の場合）
- 「認証はどのような方式を想定していますか？（メール+パスワード、SSO、OAuth等）」
- 「稼働率の目標はありますか？」
- 「CI/CDパイプラインの構成に希望はありますか？」

## フェーズ6: 技術要件

**ゴール**: 技術スタックの希望・制約を確認する。未定でもブロッカーにしない。

確認観点:
- 技術スタックの希望・制約
- 外部サービス連携
- インフラ環境（クラウド、オンプレ）

質問例:
- 「技術スタックに希望や制約はありますか？まだ決まっていなければ未定で進めます」
- 「外部のAPIやサービスと連携する必要はありますか？」
- 「インフラはAWS、GCP、Azureなどどれを想定していますか？」

## 質問の終了判断

以下の条件を**すべて**満たしたとき、ヒアリング完了と判断する:
1. プロジェクトの目的・背景・構成要素が明確
2. 利用主体（人間ユーザー / 外部システム / スケジューラ等）と主要な利用シーンが特定されている
3. 機能の全体像が洗い出され、構成要素との対応と優先度が付けられている
4. Must機能について以下が定義できるレベルの情報がある:
   - 共通: I/O・バリデーション・エラーハンドリング・データ永続化
   - UIを含む機能: 画面要件（レイアウト、遷移、インタラクション）
   - APIを含む機能: エンドポイント、認証、レート制限
   - バッチを含む機能: 実行トリガー、リトライ、冪等性
   - 業務自動化を含む機能: 対象システム、操作手順、前提条件、失敗時の再開方法、人間の介入ポイント
5. 非機能要件が構成要素に応じて確認済み（性能・セキュリティ・可用性・運用）
6. 技術スタックは確認済み（未定でも可）

条件を満たしたら「ヒアリング内容をもとに要件定義書を作成します。よろしいですか？」と確認してから生成に移る。
