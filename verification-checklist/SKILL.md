# Verification Checklist Generator

仕様書とAI実装済みコードを照合し、人間が行うべき動作確認チェックリストをMarkdownで出力する。

## ワークフロー

1. 入力の特定（仕様書・実装範囲の確定）
2. 仕様書と実装コードの読み込み
3. 差分分析と検証項目の抽出
4. チェックリストMarkdown生成

## Step 1: 入力の特定

ユーザーから以下を確定する（明示されていなければ必ず確認する）:

- **仕様書の場所**: ファイルパス、URL、Issue番号等。形式は問わない（Markdown、Google Docs、Notion、Figma等）
- **実装コードの範囲**: 以下のいずれかで確定する
  1. ユーザーが明示したディレクトリ or ファイル群
  2. ユーザーが指定したコミット範囲（例: `git diff main..feature-branch`）
  3. ユーザーが指定したブランチ（例: `git diff main...HEAD`）

### 実装範囲の確定ルール

- ユーザーが範囲を明示しない場合、**推測せずに確認する**。「どのコミット/ブランチの変更が対象ですか？」と聞く
- `git diff` や `git status` を無条件に使わない。dirty worktreeや並行作業の変更を誤って拾うため
- 範囲が確定したら、対象ファイル一覧をユーザーに提示して合意を取る

## Step 2: 分析

仕様書と実装コードを読み込み、以下の観点で分析する。

### 懸念箇所の判定基準

懸念はコードと仕様から**観測可能な事実**に基づいて判定する。「AIの直感」ではなく、第三者が同じコードを読んでも同じ結論に至る客観的根拠を持つこと。

判定の根拠となる観測:
- 仕様の記述が曖昧で、実装が特定の解釈を選択している（仕様の該当箇所を引用できること）
- 条件分岐が多い、状態遷移が複雑（具体的な分岐数や状態数を示せること）
- 外部依存がある（API、DB、ファイルI/O等、具体的な依存先を示せること）
- エラーハンドリングが不足している可能性（具体的にどのエラーパスか示せること）
- パフォーマンスに影響しうる処理（N+1、大量ループ等、具体的なパターンを示せること）
- セキュリティに関わる処理（認証・認可・入力検証、具体的な脆弱性パターンを示せること）

### 抽出する観点

1. **仕様カバレッジ**: 仕様書の各要件が実装されているか
2. **懸念箇所**: 上記の判定基準に該当する箇所
3. **仕様と実装の乖離**: 仕様に書かれているが未実装、または仕様にない実装
4. **エッジケース**: 境界値、空入力、大量データ、同時実行等
5. **UI/UX確認点**（フロントエンドの場合）: レイアウト、レスポンシブ、アクセシビリティ

### 記述の原則: 事実のみ書く

- 「なぜ実装できなかったか」「なぜ追加したか」などの推測・理由づけは書かない
- 代わりに観測事実を書く: 「仕様の§3.2に記載があるが、対応するコードが見当たらない」「仕様に記載がないが `{関数名}` が追加されている」
- git履歴や会話ログを読まない限り確証が持てない情報は書かない

## Step 3: チェックリスト生成

[references/template.md](references/template.md) のテンプレートに従い、チェックリストMarkdownを生成する。

### 出力ルール

- **出力先**: デフォルトは会話内にMarkdownとして出力。ユーザーが「ファイルに書いて」「VERIFICATION.mdに保存して」と言った場合のみファイルに書き出す
- 各チェック項目にはチェックボックス `- [ ]` を付ける
- 懸念度を3段階で表示: `[!!!]` 高 / `[!!]` 中 / `[!]` 低
- 全項目に**前提条件**と**期待結果**を含める（確認者が合否判定できるようにする）
- 懸念度が高い項目には根拠（観測事実）を添える

### 仕様参照の書き方

仕様の媒体に応じて適切な参照形式を使う:
- Markdownファイル → セクション見出し or 行番号（例: `SPEC.md > §3.2 認証フロー`、`SPEC.md:45-52`）
- Google Docs / Notion → セクション見出し（例: `SPEC > 認証フロー`）
- GitHub Issue → Issue番号と該当部分の引用（例: `#123 "ログイン時に〜"`）
- Figma → ページ/フレーム名（例: `Figma > Login Screen > Error States`）
- 行番号やURLが取得できない場合はセクション見出しのみで可。**参照不能な場合に架空の参照を作らない**

### 実装参照の書き方

- 実装ファイルパスと行番号を記載: `src/auth.ts:42`
- 行番号が特定できない場合はファイルパスと関数/クラス名: `src/auth.ts > login()`

### 懸念度の判定基準

- `[!!!]` 高: 仕様が曖昧で複数解釈可能 / セキュリティに直結 / データ損失リスク / 複雑な状態管理
- `[!!]` 中: 外部依存あり / エッジケース未カバーの可能性 / パフォーマンス懸念 / エラーハンドリング不足の可能性
- `[!]` 低: 仕様通りだが念のため目視確認推奨 / UIの微調整が必要かもしれない

## Step 4: 出力と説明

チェックリストを出力した後、以下を簡潔に報告する:

- 検証項目の総数
- 懸念度別の内訳
- 特に注意すべき箇所のサマリ（上位3件程度）
