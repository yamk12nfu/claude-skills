# Figma Make → 要件定義書

Figma MakeのプロトタイプURLを起点に、Figma MCPやユーザーとの対話で設計情報を収集し、対話的ヒアリングで要件を確定させ、SPEC.mdを生成する。

## URLの判定

- **対象**: `figma.com/make/` を含むURL
- **対象外**: `figma.com/design/`, `figma.com/board/` 等。これらは通常のFigmaファイルであり、このスキルの対象ではない

## ワークフロー

```
1. URL受領・プロトタイプ情報の収集
2. 収集結果の共有と初期ヒアリング
3. セクション別の深掘りヒアリング（確定まで繰り返し）
4. 要件定義書（SPEC.md）生成
5. レビュー・修正
```

## Step 1: URL受領・プロトタイプ情報の収集

ユーザーからFigma MakeのURLを受け取り、プロトタイプの情報を収集する。

### Figma MCPのMakeファイル制約

- Makeファイルで使用可能なMCPツールは **`get_design_context` のみ**
- `get_metadata`、`get_screenshot` はDesignファイル専用でありMakeには使用できない
- `get_design_context` はFigma URL内のノード情報をもとに対象を特定する。Makeファイル内のノード一覧を自動列挙する手段はない
- 取得できる情報の範囲や粒度は、渡すURLやノード指定に依存し、プロトタイプ全体を一括で網羅できる保証はない

### 1-1. URLにノード/フレーム情報が含まれる場合

URLにノードIDやフレーム指定が含まれている場合（例: URL末尾に `?node-id=...` がある、または特定フレームへのリンク）、Figma MCPの `get_design_context` を呼び出して情報を取得する。

- 取得できた情報（レイアウト、UI要素、テキスト等）を整理する
- 取得結果はあくまで「MCP経由で確認できた範囲」として扱い、プロトタイプ全体を網羅したと仮定しない
- 他の画面やフレームの情報が必要な場合、ユーザーにFigma上で対象フレーム/レイヤーのリンクをコピーして共有してもらい、追加で `get_design_context` を呼び出す

### 1-2. URLだけでは対象を特定できない場合

素の `figma.com/make/{fileKey}/{name}` URLにはノード指定が含まれないため、`get_design_context` で何が返るか保証できない。以下の順で情報を収集する:

1. ユーザーにFigma上で主要フレーム/レイヤーのリンクをコピーして共有してもらい、そのURLで `get_design_context` を呼び出す
2. リンク共有が難しい場合、ユーザーにプロトタイプのスクリーンショット提供を依頼する
3. 上記が難しい場合、URLをブラウザ（Playwright）で開いてスクリーンショット取得を試みる
4. ユーザーにプロトタイプの画面構成・機能を口頭で説明してもらう

いずれの手段でも、収集できた情報は「確認できた範囲」に限定して扱い、網羅性を仮定しない。

## Step 2: 収集結果の共有と初期ヒアリング

収集した情報を整理し、**観測事実と推測を区別して**ユーザーに提示する:

- **確認できた主要UI構成**（ボタン、フォーム、リスト等。MCP/スクリーンショットから直接読み取れたもの）
- **確認できたビュー/画面候補**（取得範囲内で認識できたもの。網羅性は保証しない旨を添える）
- **推測される主要フロー候補**（※仮説として提示。確認を取る）

提示後、以下を確認（3問以内）:

1. 「このプロトタイプは何を実現するためのものですか？」（背景・目的）
2. 「確認できた画面・機能に過不足はありますか？見落としている画面や機能があれば教えてください」
3. 「今回の実装スコープと想定ユーザーを教えてください」

## Step 3: セクション別の深掘りヒアリング

SPEC.mdの各セクションについて順番にヒアリングする。**各セクションの内容をユーザーに提示し、確認を取ってから次に進む。**

### ヒアリングの原則

- **一度に質問は3つまで**。大量の質問で圧倒しない
- **UIから直接読み取れる事実**（画面名、ボタン名、フォーム項目等）と、**UIから推測した仮説**（ビジネスロジック、API構成、データ構造等）を明確に区別する
- 事実 → 「Figma分析から以下が確認できました」として提示
- 仮説 → 「以下はプロトタイプからの推測です。正しいですか？」として確認を求める
- **曖昧な回答は具体化するまで深掘り**する（「適切に」→「具体的にはどのように？」）
- **各セクション完了時に内容を提示**し「この内容で確定してよいですか？」と確認する
- ユーザーが「わからない」「おまかせ」と答えた場合、ベストプラクティスに基づく提案をして確認を取る
- **未確認の仮説は決してSPEC.mdに確定事項として書かない**。確認が取れないものは「未確定事項」セクションに記載する

### 3-1. 概要

Figma分析から推測した概要を**仮説として**提示し確認:
- 対象機能・課題
- 背景
- 今回解決したいこと

### 3-2. ゴール / 非ゴール

- 実装で達成すること（ゴール）
- 今回やらないこと（非ゴール）を明確化
- プロトタイプにあるが実装しない機能があれば非ゴールに分類

### 3-3. 技術スタック

- 使用する言語/フレームワーク/ライブラリ
- インフラ構成
- 採用理由
- ユーザーの技術的な好みや制約を確認
- ※UIからは推測できない領域。必ずユーザーに直接確認する

### 3-4. アーキテクチャ

- 構成の責務分割
- 主要コンポーネント間のデータ/制御フロー
- 画面構成をベースに**提案として**提示し確認

### 3-5. API設計

- エンドポイント/メソッド/入出力
- エラーハンドリング方針
- 認証・認可方針
- フォームやアクションから**仮説として**エンドポイントを推測し、ユーザーに確認

### 3-6. データモデル

- エンティティ、主要フィールド、制約
- リレーション
- 永続化方針
- リスト/テーブル/フォームから**仮説として**データ構造を推測し、ユーザーに確認

### 3-7. UI/UX方針

- 画面要件（Figma分析で確認できた範囲を事実として提示し、不足分を確認）
- 操作フロー・状態遷移
- レスポンシブ対応方針
- アクセシビリティ方針

### 3-8. 非機能要件

- 性能要件
- セキュリティ要件
- 信頼性・可用性
- 運用・監視方針
- テスト方針

## Step 4: SPEC.md生成

全セクションが確定したら、`references/spec-template.md` に従って要件定義書を生成する。

- 出力先はユーザーに確認する（デフォルト: プロジェクトルートの `SPEC.md`）
- 未確定事項は「未確定事項」セクションに明示的にまとめる
- 仮説のまま確認が取れなかった項目は決して確定事項として記載しない

## Step 5: レビュー・修正

生成した要件定義書をユーザーに提示し、修正点を確認する。修正があれば反映し、再度確認を取る。「問題ない」「OK」の回答を得るまで繰り返す。
